# 테스트 결과 재평가와 비교 보고서 개선 경험

## 1. Overview

이 작업은 자연어 SQL 워크플로우 테스트 결과를 단순 실행 성공률이 아니라 업무 의도 충족 관점에서 다시 평가한 경험입니다. 기존 결과는 쿼리가 생성되었는지와 실제로 자연어 요청을 만족했는지가 섞여 있었고, 연계 질문은 여러 step의 품질을 하나의 case 결과로 뭉뚱그릴 위험이 있었습니다.

포트폴리오 관점에서 핵심은 “숫자를 다시 만든 것”이 아니라, 평가 기준을 재정의하고 raw 결과를 다시 읽어 근거 기반으로 재채점한 과정입니다. 이후 모델별, 날짜별, run별 그래프와 보고서를 재작성하면서 어떤 모델이 쿼리를 잘 만드는지와 어떤 모델이 업무 의미를 잘 지키는지를 분리해 볼 수 있게 만들었습니다.

## 2. 문제 발생

초기 비교 결과는 Query PASS와 Semantic PASS가 명확히 분리되지 않아 해석이 어려웠습니다. 쿼리 후보가 생성되고 preview까지 가능하면 테스트가 성공처럼 보였지만, 실제 샘플 row와 SQL을 함께 보면 자연어 요청의 의미를 만족하지 못하는 경우가 있었습니다.

특히 다음 문제가 반복되었습니다.

- Query PASS가 높아도 Semantic PASS가 낮은 경우가 많았습니다.
- 연계 질문에서 한 step은 맞지만 뒤 step이 선행 결과를 반영하지 못하는 경우가 있었습니다.
- “보정 후 다시 계산”, “각 매체별로 분리”, “결과를 바탕으로 집계”처럼 순서와 의존성이 중요한 요청이 단일 case 결과로 접힐 위험이 있었습니다.
- 기존 보고서가 raw/analysis 파일 링크 중심으로 보일 수 있어 포트폴리오나 의사결정 자료로 쓰기 어려웠습니다.

이 문제를 해결하려면 테스트 산출물을 다시 실행하는 것보다, 이미 생성된 raw 결과를 직접 검토해 평가 기준을 바로잡는 것이 먼저였습니다.

## 3. 사용자 지시와 제약 조건

작업 지시는 명확했습니다. raw markdown 결과를 순서대로 다시 읽고 Query PASS와 Semantic PASS를 직접 판단해야 했습니다. 여기서 Query PASS는 쿼리가 실행 또는 preview 가능한지를 뜻하고, Semantic PASS는 SQL과 샘플 row가 자연어 요청에 맞는지를 뜻했습니다.

중요한 제약은 다음과 같았습니다.

- raw DB는 수정하지 않습니다.
- 테스트나 LLM을 다시 실행하지 않고, 기존 raw 결과를 근거로 판단합니다.
- 연계 질문은 하나의 case로 합치지 않고 step별로 평가합니다.
- 기존 비교 문서와 오래된 그래프는 제거하고 새 기준으로 다시 작성합니다.
- 보고서는 파일 링크 나열이 아니라 통계, 그래프, 해석 중심으로 작성합니다.
- 그래프는 모델별, run별 비교가 가능해야 하고 제목, 라벨, 범례가 겹치지 않아야 합니다.

이 제약 덕분에 해결 방향이 분명해졌습니다. 목적은 모델을 다시 호출하는 것이 아니라, 평가 기준을 재정의하고 산출물을 감사 가능한 형태로 재구성하는 것이었습니다.

## 4. 해결 과정

### 4.1 평가 기준 분리

먼저 Query PASS와 Semantic PASS를 분리했습니다. Query PASS는 SQL 후보 생성, validation, preview 가능 여부를 중심으로 판단했습니다. 반면 Semantic PASS는 자연어 요청, SQL, 샘플 row가 서로 맞는지 직접 확인했습니다.

이 분리는 중요했습니다. 예를 들어 쿼리가 문법적으로 맞고 preview도 가능하더라도, 요청한 metric이 아닌 다른 metric을 보정하거나, 집계 요청인데 샘플 결과가 없거나, count 요청인데 COUNT가 없는 경우는 Semantic FAIL로 봐야 했습니다.

### 4.2 연계 질문의 step 단위 평가

연계 질문은 단일 case 결과만으로는 품질을 설명할 수 없었습니다. 그래서 case-level semantic에는 `step_evaluated`를 두고, 각 요청 묶음마다 Step Query PASS와 Step Semantic PASS를 따로 기록했습니다.

이 방식은 다음 차이를 드러냈습니다.

- 전체 case는 Query PASS라도 일부 step만 semantic fail일 수 있습니다.
- 선행 보정 step은 맞지만 후속 집계 step이 선행 결과를 반영하지 않을 수 있습니다.
- 독립 step과 dependent step을 같은 방식으로 평가하면 실제 오류 원인을 놓칠 수 있습니다.

### 4.3 재생성 스크립트와 검증 루프

raw markdown을 다시 읽고 결과 라벨을 재기입하는 일회성 재생성 스크립트를 만들었습니다. 이 스크립트는 raw 결과를 읽어 `Result`, `Semantic`, `Step Query Result`, `Step Semantic` 라벨을 새 기준으로 다시 썼고, 분석 markdown과 비교 보고서, 그래프를 생성했습니다.

중간 검증에서 중요한 버그도 발견했습니다. 샘플 row에 `after.<metric> = 0` 값이 있는데도 markdown table parsing이 이를 놓쳐 semantic fail로 잘못 판단하는 문제가 있었습니다. 이 문제를 수정하면서 “평가 자동화도 검증 대상”이라는 점을 확인했습니다.

### 4.4 Linked Coherence PASS 추가

추가로 연계 질문이 서로 잘 이어지는지를 평가하는 `Linked Coherence PASS`를 도입했습니다. 이 지표는 step이 개별적으로 맞는지만 보는 것이 아니라, 선행 보정/파생 이후 후속 조회나 집계가 같은 매체, metric, 조건 흐름을 유지하는지 판단합니다.

Linked Coherence는 다음 기준으로 평가했습니다.

- step들이 자연어 요청의 순서를 유지하는가
- dependent step이 선행 step 결과를 전제로 하는가
- 같은 매체와 metric 흐름이 중간에 깨지지 않는가
- 각 step의 semantic이 통과했는가

이 지표를 추가하면서 연계 질문의 문제를 더 명확히 볼 수 있었습니다. Query PASS가 거의 포화 상태여도 Linked Coherence가 낮으면 실제 업무 흐름은 아직 불안정하다는 뜻입니다.

## 5. 결과

최종 산출물은 Query PASS, Case Semantic PASS, Step Query PASS, Step Semantic PASS, Linked Coherence PASS를 분리해 보여줍니다. 또한 모델별, 날짜별, run별 그래프를 생성해 성능 차이를 시각적으로 비교할 수 있게 했습니다.

핵심 결과는 다음과 같습니다.

- Query PASS는 대부분 매우 높았습니다.
- 실제 차이는 Step Semantic PASS와 Linked Coherence PASS에서 나타났습니다.
- 최신 테스트 결과가 과거 결과보다 의미 충족률과 연계 흐름 유지 측면에서 개선되었습니다.
- 관리형 API 모델은 semantic 안정성에서 더 강한 개선을 보였고, 로컬 양자화 모델은 보안성과 운영 통제 측면에서 장점이 있었습니다.

보고서에는 모델 운영 장단점도 추가했습니다. 관리형 API는 유지보수와 품질 안정성 측면에서 유리하지만 외부 의존성과 비용 누적이 있습니다. 로컬 양자화 모델은 데이터 통제와 오프라인 운영에 강하지만 GPU 운영, 장애 대응, 품질 튜닝 부담이 큽니다.

## 6. 배운 점

이번 작업의 가장 큰 교훈은 “실행되는 SQL”과 “업무적으로 맞는 SQL”은 다르다는 점입니다. 자연어 기반 SQL workflow에서는 문법 검증만으로는 충분하지 않습니다. 샘플 row, preview, 후속 step의 dependency까지 함께 검증해야 합니다.

또한 모델 비교는 단일 PASS율보다 평가 축을 분리해야 의미가 있습니다. Query PASS는 SQL 생성 능력, Semantic PASS는 업무 의도 충족 능력, Linked Coherence PASS는 연계 업무 흐름 유지 능력을 각각 보여줍니다.

## 7. Public Safety Note

이 문서는 공개 포트폴리오용으로 작성되었으며 실제 원천 파일명, 운영 DB 이름, 민감한 샘플 row, 고객 식별 정보, 내부 프롬프트를 포함하지 않습니다. 모든 설명은 workflow 설계와 평가 방법 중심으로 일반화했습니다.
